# PDFYandexVisionBlockV1 — Документация проекта

## 1. Общее описание

**PDFYandexVisionBlockV1** — это интеллектуальный блок для платформы визуального программирования (Google Blockly / PuzzleBot), предназначенный для автоматического распознавания и извлечения структурированных данных из PDF-документов через API **Yandex Vision OCR**.

Блок принимает на вход PDF-файл, отправляет его в Yandex Vision, получает результат распознавания и извлекает из текста **реквизиты** (номер, дату, сумму, ИНН, КПП, контрагентов, товарные позиции и пр.), возвращая их в структурированном виде — как словарь Python или JSON-строку.

### Ключевая особенность

В отличие от простого OCR-распознавания, блок **понимает семантику документа**: он не просто возвращает сырой текст, а определяет тип документа (платёжка, счёт-фактура, накладная, договор...), находит и связывает между собой реквизиты, разделяет контрагентов на поставщика и покупателя, извлекает табличные данные о товарах/услугах.

### 2.1 `block.json` — Интерфейс блока

Определяет визуальное представление блока в редакторе Blockly:
- **5 входов (inputs):**
  1. `TOKEN` — токен доступа к Yandex Cloud (String)
  2. `FOLDER_ID` — идентификатор каталога в Yandex Cloud (String)
  3. `FILE_PATH` — путь к PDF-файлу (String)
  4. `LANGUAGE` — выпадающий список: `смешанный` / `русский` / `английский`
  5. `OUTPUT_FORMAT` — выпадающий список: `Словарь (dict)` / `JSON-строка`
- Цвет блока: оранжевый (`#F9A825`)
- Подсказка: «Распознает PDF через Yandex Vision и извлекает реквизиты»

### 2.2 `src/__init__.py` — Бизнес-логика

Содержит всю логику распознавания и извлечения данных. Ключевые компоненты описаны ниже.

## 6. Стратегия аутентификации в Yandex Vision

Блок поддерживает **три схемы аутентификации** и пробует их последовательно:

| # | Схема             | Тип токена           | Аутентификация              |
|---|-------------------|----------------------|-----------------------------|
| 1 | `Api-Key`         | Сервисный API-ключ   | `Authorization: Api-Key X`  |
| 2 | `Bearer`          | IAM-токен            | `Authorization: Bearer X`   |
| 3 | OAuth → IAM       | OAuth-токен → обмен  | `POST /iam/v1/tokens`       |

Каждая схема пробуется с `DOCUMENT_RECOGNITION`, а при ошибке `"invalid value"` — с fallback-режимом `TEXT_DETECTION`.

---

## 7. Стратегия извлечения дат

Умный поиск даты документа — не просто первая найденная дата, а контекстный анализ:

1. Если дата одна — возвращается она
2. Если дат несколько — ищется дата рядом с ключевыми словами:
   - «дата», «дата платежа», «дата документа»
   - Контекст рядом с «платёжное поручение»
   - Контекст рядом с «№ платежа», «№ плат.», «номер поруч.»
3. Если контекст не найден — возвращается **последняя** дата в документе

---

## 8. Стратегия поиска номера документа

Сложный многоэтапный алгоритм для платёжных поручений:

1. **Хвостовой поиск** — после последней даты в документе
2. **Меж-датный поиск** — между двумя последними датами
3. **Контекстный поиск** — рядом со словами «платёжное поручение», «поручение»
4. **Общий поиск** — любое 5-10 значное число (не являющееся датой)

---

## 9. Обработка ошибок

Блок использует иерархию кастомных исключений:

```
RuntimeError
└── DocumentRecognitionError
    ├── YandexVisionAuthError        # 401/403
    ├── YandexVisionQuotaError       # 429
    ├── YandexVisionNetworkError     # Сетевые ошибки / 5xx
    └── YandexVisionResponseError    # Некорректный JSON/структура
```

При любой ошибке возвращается **graceful fallback** — структура с `ok: false`, `error: "описание"` и заполненными `null`-полями. Пользователь видит нативный диалог ошибки (macOS — AppleScript, Windows — MessageBox, Linux — tkinter).

## 14. Ключевые архитектурные решения

1. **Graceful degradation** — если DOCUMENT_RECOGNITION недоступен, автоматический fallback на TEXT_DETECTION, затем на локальный PyPDF2
2. **Triple-auth** — три схемы аутентификации перебираются автоматически
3. **Контекстно-зависимый парсинг** — даты, номера и суммы извлекаются с учётом окружающего контекста, а не просто по порядку
4. **Типизация через dataclasses** — `ImportantData`, `DetailsData`, `ItemData`, `ExtractedDocument` со `slots=True` для эффективности
5. **Русскоязычный API** — ключи результата на русском (`важное`, `данные`, `метаданные`, `товары`), что упрощает интеграцию в русскоязычные RPA-системы
